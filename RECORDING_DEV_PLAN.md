# 目标

把“录制”从“偶尔能用的按钮”提升为可预期、可解释、跨浏览器可用的功能，并预留后续云端录制与转码能力。

# 当前已实现（本轮交付）

- 录屏兼容性：`getDisplayMedia({ video: true, audio: true })`
- 新增二选一：录屏 vs 合成录制（会议画面 canvas 合成）
- 录制中统计：显示时长与大致文件大小（基于已写入 chunks 累计）
- 输出格式：`webm`（浏览器原生 MediaRecorder）

# 近期优化（前端即可完成）

## 1) 合成录制增强

- 布局：对齐当前 UI（全屏时只录当前 tile；网格时按排序顺序合成）
- 水印：可选在 canvas 角落绘制（会议号、时间戳、品牌 logo）
- 片头/片尾：开始前渲染 1-2s 的标题卡（会议名、参与者数量）
- 音频策略：
  - 默认混音：本地 + 远端音频混到一个轨道
  - 选项：仅录本地 / 仅录远端 / 全部混音（UI 可扩展）

## 2) UI 体验

- 录制前提示：格式、隐私与权限说明（尤其是 macOS 屏幕录制权限）
- 录制后管理：保存文件名可编辑，或弹出“保存/取消”

# 中期优化（需要服务器，但可先不做）

## 1) MP4 转码（服务端）

MediaRecorder 在多数浏览器输出 `webm`，Safari/部分环境对 `webm` 支持不一致。要统一成 `mp4`，建议走服务端转码：

- 前端上传 webm → 后端 `ffmpeg` 转码 → 返回 mp4 下载/存储地址
- 对服务器资源要求（粗略经验值，实际与编码参数/硬件有关）：
  - 720p/30fps：约 0.5–1 vCPU，内存 200–500MB，磁盘 I/O 中等
  - 1080p/30fps：约 1–2 vCPU，内存 500MB–1GB
  - 并发转码需要线性扩容；建议队列化（BullMQ/Redis）或分离转码 worker
  - 若有 GPU（NVENC/VAAPI/VideoToolbox）可显著降低 CPU 占用

建议：先做“上传 + 后端落盘”，再逐步接入转码队列。

## 2) 直传服务器（归档/审计）

- 前端录制完成后上传（分片上传/断点续传）
- 后端存储（S3/MinIO/OSS）+ 元数据（会议号、开始结束时间、参与者）

# 远期规划（云端录制 / SFU）

P2P WebRTC（当前架构）无法在服务器端“直接拿到所有参会者音视频并混流录制”。要实现真正的云端录制，通常需要：

- 引入 SFU（Janus / mediasoup / LiveKit / Jitsi / Pion 等）
- 服务器侧订阅各路音视频 → 混流（或多路录制）→ 存储
- 可选：服务端合成布局、字幕、AI 纪要

这部分属于架构升级，建议单独立项：

- 阶段 1：先引入 SFU 做转发（降低 P2P 连接数爆炸）
- 阶段 2：加录制 worker（合成/多轨）
- 阶段 3：完善权限、存储、回放、审计链路
